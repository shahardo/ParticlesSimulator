## Context

The user wants to build a 3D physics simulator of up to N=1,000,000 particles, as a **learning project** (new to both TypeScript and GPU/compute programming), delivered in 5 incremental stages of increasing physical realism:

1. Gravity-only N-body attraction
2. + elastic collisions
3. + inelastic collisions (adjustable restitution)
4. + per-particle heating and radiative cooling
5. + a second collisionless species (dark matter) alongside baryonic matter

The simulator must also display live derived statistics (average enclosing sphere radius, average rotation axis + angular speed), look visually polished with easy 3D camera navigation, and let the user toggle GPU use on/off.

Because clarity and incremental buildability were explicitly prioritized over raw performance, every design choice below favors the simpler, more learnable option when it's still fast enough to hit the target scale.

## Tech stack (decided with user)

- **TypeScript + Vite + Three.js `0.185.1` (r185)**, using **`WebGPURenderer`** and **TSL** (Three.js Shading Language — a JS-like node API that compiles to real WGSL compute shaders) for **both** physics compute and rendering. Keeping compute and render in one WebGPU device means particle buffers are shared directly with zero CPU round-trip for display.
- **Controls library: Tweakpane** (supports folders + read-only "monitor"/graph bindings, a good fit for the live stats readouts).
- Camera navigation: Three.js `OrbitControls`.
- Post-processing: `THREE.RenderPipeline` (r183+ name for the old `PostProcessing` class) + TSL `bloom()`.
- All of the following were confirmed directly against live Three.js source/examples (not guessed): `Fn()`, `instancedArray()`, `instanceIndex`, `struct()`, `atomicAdd`/`atomicStore`/`atomicLoad` (explicitly WebGPU-only, no WebGL2 compute fallback exists), `renderer.compute()`/`computeAsync()`, `renderer.getArrayBufferAsync()` for CPU readback, and the `SpriteNodeMaterial` + `instancedArray(...).toAttribute()` pattern for zero-CPU-touch instanced particle rendering.
- **GPU/CPU is a hard binary split, not a 3-tier fallback**: WebGPU compute shaders/atomics have no WebGL2 equivalent, so the CPU backend is a separately hand-written TypeScript implementation of the same algorithm, not "the same shader running slower." Detect `navigator.gpu` at startup and force/gray-out to CPU-only if absent.

## Architecture

A `SimulationBackend` interface (`init(N)`, `step(dt)`, `setParams(...)`, `getSnapshot()`, `dispose()`) is implemented twice — `GpuBackend` (TSL compute kernels) and `CpuBackend` (plain TS, optionally Web-Worker-parallelized later). The renderer and UI only ever talk to this interface, so switching GPU↔CPU at runtime doesn't ripple through the rest of the app.

**Shared spatial structure — two uniform grids, rebuilt every frame, reused across every stage:**
- **Fine grid**: sized so each cell holds ~4 particles on average (`cellsPerAxis ≈ cbrt(N/4)`). Used for near-field direct gravity (own cell + 26 neighbors) and *all* collision neighbor queries.
- **Coarse grid**: originally planned as a small fixed resolution independent of N (default 16³ = 4,096 cells). **As built (M4), this changed** — see below.
- Both grids are built via the same 5-pass counting-sort recipe: clear → count per cell → prefix sum → scatter into sorted-index buffer → per-cell mass/center-of-mass aggregate. The coarse grid's aggregates are derived from blocks of fine-cell aggregates rather than binning a second time.
- Simulation domain is a **fixed bounding box with soft bounce walls** on all 6 faces — avoids needing to dynamically resize/rebuild the grid domain. (As built: the box radius is a live-adjustable Tweakpane slider, not a fixed constant — resizing it just changes the wall-bounce boundary and the visual grid helper, no reinit needed.)

### As-built deviations (discovered during M4)

The original plan's fixed-16³-coarse-grid idea doesn't work at CPU-scale N: at N=50,000 that's 50,000 × 4,096 ≈ 200M monopole evaluations/frame, far too slow single-threaded. The as-built `computeGridSizing()` instead derives the coarse resolution from N (`coarseCellsPerAxis ≈ cbrt(N/4)/blockSize`), keeping far-field cost roughly proportional to N rather than fixed — see `src/sim/cpu/grid/UniformGrid.ts`.

Two real bugs surfaced while building this, both worth remembering before the M6 GPU port re-implements the same algorithm:

1. **`blockSize=4` (fine cells per coarse cell) left a coverage gap.** The near-field neighborhood only reaches 1 fine cell in every direction (a 3×3×3 window), which doesn't fully cover a 4-wide coarse block — mass in the gap was invisible to both near-field (too far) and far-field (excluded as the particle's "own" cell). Measured as ~40% average force error vs. an exact brute-force reference. Fixed by using `blockSize=2`, which mathematically guarantees the near-field window always fully covers the particle's own coarse block.
2. **Far-field momentum conservation vs. spatial resolution is a real tension, and the first fix traded one bug for another.** Using each particle's *exact* position against a distant cell's monopole isn't momentum-conserving (particles sharing a cell don't pull equally hard on a distant cell, so the reaction back doesn't balance — compounds every step into unbounded energy injection; invisible over a few seconds, obvious as steady RMS-radius growth over 60+ simulated seconds). The seemingly-obvious fix — use the particle's *own coarse cell's center of mass* instead, which *is* exactly momentum-conserving between any two cells — traded that bug for a different one: every particle sharing a cell now gets an identical far-field vector, so the cloud visibly fragments into one clump per coarse cell (worst at the grid's corner cells). The actual fix keeps exact-position far-field (smooth per-particle variation, no fragmentation) and separately cancels the net momentum bias with a cheap global correction after the main loop (sum of `m·a`, then subtract per unit mass). See the long comment in `src/sim/cpu/physics/gravity.ts` for the full reasoning, and `gravity.test.ts` for the regression test.

Both bugs are the kind that look fine in a quick check and only show up over a longer run or in a corner case — worth a real screenshot a few seconds into playback *and* a numeric energy trace when touching this code again, not just one or the other.

### As-built deviations (discovered during M5)

- **`typescript@7.x` is unusable with this codebase.** The new native Go-based compiler (`typescript-go`) hangs for 5-10 minutes and then crashes with an out-of-memory fatal error when type-checking TSL's heavily-chained/proxied node API (confirmed via the crash's own stack trace — deep recursive type-relation checking hundreds of frames deep inside the Go checker). This didn't show up until M5 because the CPU backend never touches TSL types at all. **`typescript` is pinned to `^6.0.3`** (the last pre-native-rewrite major line) in `package.json` — do not bump past 6.x while this project uses TSL, unless a future TS7 release is confirmed to fix this. Also: prefer `./node_modules/.bin/tsc.cmd` over `npx tsc` in this repo — `npx tsc --version` alone was independently observed to hang too (separate overhead on top of the above).
- **`SimulationBackend.getSnapshot()` doesn't fit GPU-resident data.** Reading GPU buffers back into a CPU `Float32Array` synchronously every frame would defeat the entire point of GPU compute. `GpuBackend.getSnapshot()` returns an empty stub; the real GPU render path (`GpuParticlePoints`) reads the TSL position buffer directly via `getPositionsNode()` instead, bypassing `getSnapshot()` entirely. Added a `kind: 'cpu' | 'gpu'` discriminant to `SimulationBackend` so render/debug code can branch cleanly. `GpuBackend.debugReadPositions()` (async, via `renderer.getArrayBufferAsync()`) exists purely for manual/test verification, never called from the render loop.
- **TSL's node types aren't fully expressible even where `@types/three` tries** (e.g. `.toAttribute()` exists but composing return types with `ReturnType<>` is what triggered the TS7 hang above). GPU-side buffer/kernel handles are typed `any` in `GpuBackend`/`GpuParticlePoints` rather than fought into precise types — deliberate, not an oversight.
- **A "Use GPU" toggle was added to the panel now**, ahead of M7, since some way to exercise the GPU path through the actual UI (not just scripted tests) seemed worth the small cost. It's a plain manual checkbox with a console-warning safety net if no real WebGPU adapter is available (falls back to CPU and self-corrects the checkbox) — no adaptive default-N benchmarking or graying-out yet; that's still M7's job.
- **Could not visually verify the GPU compute/render path at all.** This dev environment's headless Chromium has no real WebGPU adapter (`No available adapters`, falls back to WebGL2 — see M0's note), and WebGL2 has zero support for compute shaders/storage buffers. Everything GPU-specific here (`GpuBackend`, `GpuParticlePoints`) is verified only by: type-checking cleanly, matching patterns fetched directly from the official `webgpu_compute_particles.html` example at the exact installed Three.js version (r185), and confirming the CPU path + graceful-fallback path both work correctly. **The actual GPU rendering/compute needs to be checked on real GPU hardware** — treat it as unverified until someone does.

### As-built deviations (discovered during M6)

- **GPU counting-sort has no local reference example to pattern-match against.** Unlike M5 (which matched the official `webgpu_compute_particles.html` example closely), this project's installed Three.js examples have no atomic-based grid-build kernel to check against — the closest analog, `ClusteredLightsNode.js`, avoids the problem entirely by using a fixed-capacity per-cluster loop instead of a real sort (viable there because `maxLights` is small; not viable here at N=100k-250k particles per grid). The build (`src/sim/gpu/grid/GpuUniformGrid.ts`) is a from-scratch TSL translation of the CPU 5-pass counting sort (`src/sim/cpu/grid/UniformGrid.ts`), using `atomicAdd`/`atomicStore`/`atomicLoad`/`.toAtomic()` confirmed directly against the installed `three` package's source (`node_modules/three/build/three.webgpu.js`), not an existing example. Treat it as higher-risk than M5's port until checked on real hardware.
- **Two passes are still genuinely single-threaded by design, not an oversight**: the fine-cell prefix sum (`GpuUniformGrid`'s `prefixSumKernel`) and the net-momentum reduction (`GpuBackend`'s `momentumReduceKernel`) each dispatch as a single GPU invocation that loops serially (`Loop({start,end}, ...)` over up to `fineCellCount` or `count` iterations). This mirrors the CPU version's already-serial prefix sum, which the plan's Risks section already flags as an accepted bottleneck rather than something to fix now — a parallel (Hillis-Steele/Blelloch) prefix sum and reduction are the same documented future optimization on both backends. At N=250,000 the momentum reduction alone is 250,000 sequential dependent memory reads on one thread; likely the single biggest per-step cost here, worth profiling first if/when M6's GPU path is ever benchmarked for real.
- **`blockSize` is assumed to be exactly 2 and unrolled in plain JS at kernel-build time** (both the coarse-cell aggregate's 2x2x2 fine-cell loop and the gravity kernel's 3x3x3 near-field neighborhood), rather than a dynamic TSL `Loop`, since `computeGridSizing` (shared with the CPU backend) always produces `blockSize=2` today. If that ever changes, both `GpuUniformGrid.ts` and `gpuGravity.ts` need updating in lockstep, not just the CPU-side constant.
- **Particle mass is hardcoded to `1` in every GPU kernel**, not read from a per-particle buffer. This matches current reality, not a shortcut: `CpuBackend` also always fills its `masses` array with `1` and has no per-particle-mass UI or setter anywhere yet. A real mass buffer is expected to land with M11 (dark matter species), at which point both backends need one. *(Superseded ahead of schedule: the "vanish" wall behavior, added out of milestone sequence, needed a real GPU mass buffer to work correctly and pulled this forward — see the wall-behavior as-built section below.)*
- **TSL's type-checker choked on plain `int`/`float` node chains here in a way M5 never hit**, not just the already-documented TS7-hang issue. Composing `clamp(int(...), int(0), int(n))` produced a real (non-hanging) `tsc` overload-resolution error under TypeScript 6 -- traced to `clamp`'s overload set matching mixed int-typed arguments against an unrelated float/vec4 overload and reporting a nonsensical mismatch. Fixed by casting the specific call (`(clamp as any)(...)`), not the whole file, and by explicitly typing every TSL-node-holding local as `any` throughout `GpuUniformGrid.ts`/`gpuGravity.ts` (not just the buffer handles, which was M5's scope) -- letting inference run across long `.mul()/.add()/.div()` chains is what triggers this class of error. Confirmed via a clean `tsc -b --force` run; if a future milestone's TSL code hits a similar overload error, this is the pattern to reach for before assuming it's a real type mismatch.
- **This dev environment's headless Chromium turned out to have a real GPU behind it after all — the "no WebGPU adapter" limitation from M0/M5 is headless-mode-specific, not hardware-specific.** Running the driver with a visible (non-headless) browser window gets a genuine `renderer: WebGPU` adapter (`navigator.gpu` negotiation succeeds outside headless mode). `.claude/skills/run-particles-simulator/driver.mjs` now supports `HEADED=1` to opt into this, and `SKILL.md`'s gotcha note is updated accordingly. This unblocked real verification of M6, which previously would have been type-check-only like M5's caveat implied.
- **M6 verified on real GPU hardware via `HEADED=1`**: toggled "Use GPU" in the live UI (`window.__debug.app.getBackend().kind === 'gpu'`), then paused and manually stepped the sim (N=5,000, default G/softening/domainRadius) while sampling RMS radius and NaN count every 20 steps via `debugReadPositions()`. Result: RMS radius fell smoothly from ~2.18 to a minimum of ~0.22 around step 120 (real gravitational collapse), then rebounded back out past ~2.9 by step 280 (energy conserved, not draining or blowing up) — zero NaNs across all 300 steps. A screenshot at the point of maximum collapse (step ~118) shows a single dense clump at the center, **not** the "8 clumps in the corners" fragmentation failure mode the CPU version hit during M4 — good evidence the far-field momentum correction ported correctly, not just the near-field sum. This is the same kind of check M4's CPU port used (visual + numeric, run long enough to catch a slow-building bug), just run manually against real hardware instead of an automated brute-force-reference test.
- **A real cross-backend numeric parity harness was built** (`src/debug/verifyGpuGravity.ts`, wired onto `window.__debug`) to go beyond the RMS-radius/screenshot check above: `debugSetPositions()`/`debugReadPositions()`/`debugComputeAccelerations()` (added to `GpuBackend`) let a debug script seed *identical* positions on a throwaway `GpuBackend` (bypassing its own RNG) and diff its per-particle accelerations directly against `computeGridGravity`'s CPU output for the same positions/grid sizing/G/softening — not against the brute-force reference, which has its own expected ~10-40% approximation error that would swamp a real port bug (see `gravity.test.ts`). Bisection variants (`verifyGpuGrid`, `verifyGpuGravityStages`, `verifyGpuGravityTerms`, `dumpGravityComparison`) isolate a mismatch down to grid-build vs. gravity-kernel vs. momentum-correction, and near-field vs. far-field specifically.
- **First run of this harness found a real, serious bug — but not where expected.** `verifyGpuGravity` initially reported ~140% average / ~1500% max relative error, and bisection showed grid-build matching *exactly* while every stage touching `positions`/`accelerations` was badly wrong. Root cause: **WGSL cannot pack `vec3` tightly in storage buffers** ("WGSL does not support packed vec3 data", `WebGPUAttributeUtils`) — every `instancedArray(count, 'vec3')` buffer (positions, velocities, accelerations, `fineCellCom`/`coarseCellCom`, `netMomentumBias`) gets silently upgraded from `itemSize=3` to `itemSize=4` (one padding float per particle) the moment it's first used in a dispatch. Reading/writing these with a plain stride-3 `Float32Array` (exactly what `debugReadPositions()`/`debugSetPositions()` did) produces a garbled, progressively-shifting mix of real values, zeros, and neighboring particles' data. **The actual `GpuUniformGrid`/`gpuGravity` kernels were unaffected the whole time** — TSL's `.element()` accessor handles the padding correctly inside compiled WGSL; only hand-rolled JS-side buffer readback/writeback (i.e., debug tooling, not the physics) had the bug. New helper `src/debug/vec3Buffer.ts` (`unpadVec3`) fixes reads. The write side needed a subtler, *non-obvious* fix: writing pre-padded (stride-4) data back into an already-padded attribute double-corrupts it, because three's own `WebGPUAttributeUtils.updateAttribute()` unconditionally re-derives the padded layout from `.array` at the *original* cached item size on every subsequent update, regardless of what's currently in `.array` — so the correct fix is to always write tightly-packed (stride-3) data and let that framework path do the one real padding pass itself, confirmed by dumping the raw buffer through the exact sequence with a since-removed scratch diagnostic (see `GpuBackend.writeVec3Buffer`'s doc comment for the full reasoning).
- **After that fix, the GPU gravity kernel matches `computeGridGravity` to float32 precision** — avg/max relative error ~2e-7 / ~3e-6 at N=500 and N=2,000 (grid build, near-field, far-field, and momentum correction all independently confirmed exact via the bisection variants above). This is a much stronger result than the earlier RMS-radius/screenshot check could give — that check couldn't have caught this bug at all, since aggregating over all particles is insensitive to *which* particle a value belongs to, and a shifted/garbled-but-similar-magnitude dataset still produces a plausible-looking collapse-and-rebound trace. Worth remembering the next time a GPU port "looks right" in a coarse visual/statistical check: a per-particle numeric diff against a CPU reference is a meaningfully harder and different test, not a redundant one.
- **Performance at this milestone's actual target scale, now measurable** via a new debug-only `window.__debug.debugRebuild(count, gpu)` hook that bypassed the (at-the-time-still-fixed) N cap — real GPU hardware, one step measured via wall-clock time with a forced sync (`debugReadPositions()`) after a batch of steps to avoid measuring only command-buffer enqueue time: **~41ms/step at N=100,000**, **~294ms/step at N=250,000**. The 100k case is comfortably interactive; 250k is not (~3.4fps) — consistent with the already-documented risk that the two genuinely-serial single-thread passes (`GpuUniformGrid`'s prefix sum, `GpuBackend`'s momentum reduction) dominate at large N. Profiling which of the two is the actual bottleneck, and a parallel (Hillis-Steele/Blelloch) replacement for one or both, is the natural next optimization — not built upfront per this project's stated priorities, but no longer a documented future concern, now a documented *measured* one. This measurement is exactly what M7's adaptive benchmark below builds on.

### As-built deviations (discovered during M7)

- **`PARTICLE_COUNT_MAX` split into `CPU_PARTICLE_COUNT_MAX` (20,000, unchanged) and `GPU_PARTICLE_COUNT_MAX` (1,000,000, the project's original overall target)** in `Config.ts`. Tweakpane has no API to change a binding's `min`/`max` in place, so the Point Cloud folder's N and Box Radius bindings are disposed and re-added together (preserving their order) whenever the active backend changes — see `Panel.ts`'s `buildCloudBindings`/`setMaxParticleCount`.
- **The adaptive benchmark (`src/app/gpuBenchmark.ts`) is a real measurement, not a guess**: it builds a throwaway `GpuBackend` at a small calibration N (20,000), times 5 real `step()`s with a warm-up step discarded first (pipeline/shader compile is a one-time cost, not representative of steady-state playback), forces a GPU sync via `debugReadPositions()` (bare `step()` calls only enqueue commands — wall-clock time around them alone would measure command-submission speed, not real GPU cost), and extrapolates a default N targeting ~33ms/step (~30fps), clamped to `[PARTICLE_COUNT_MIN, GPU_PARTICLE_COUNT_MAX]`. Assumes roughly linear cost scaling with N, which slightly overestimates a comfortable N at the high end (far-field cost grows a bit worse than linear, per the existing Risk below) — deliberately not fixed with a second calibration point, since that would double startup cost for a correction that only matters well past where this estimate already lands.
- **"Start according to detection, but remember the selection between sessions"** (explicit user decision, not assumed): `src/utils/preferences.ts` persists the user's last *explicit* "Use GPU" checkbox click to `localStorage`, distinguishing "never chosen" (`null`, first-ever run) from an explicit `false`. First-ever run defaults to auto-switching to GPU (with the adaptive benchmark) if a real adapter is available, matching the project's overall N=1,000,000 goal; any stored preference always wins over auto-detection afterward, even "stay on CPU" on a GPU-capable machine. Automatic fallbacks (GPU requested but no adapter this session) deliberately do **not** overwrite a stored preference — only an actual checkbox click does, so a temporary hardware/driver hiccup on one machine doesn't clobber a preference that's fine elsewhere.
- **The "Use GPU" checkbox is now actually disabled (grayed out) when no real WebGPU adapter is available**, via `BladeApi.disabled` (confirmed present on Tweakpane 4.0.5's `BladeApi`, which `BindingApi` extends) — replacing M5/M6's placeholder ("clickable, but silently reverts with a console warning"). The revert-on-click safety net is still there defensively but should no longer be reachable through the UI.
- **Verified end-to-end on real GPU hardware** (`HEADED=1`, same technique as M6): confirmed all three real scenarios in the live app, not just unit-level pieces — (1) first-ever run (`localStorage` cleared) auto-switches to GPU with an adaptively-benchmarked N (observed N=105,000 and N=102,000 across two separate runs — a real measurement, not a hardcoded constant, hence the run-to-run variance), checkbox shows checked, N slider's max is 1,000,000; (2) manually unchecking "Use GPU" while at N=105,000 correctly re-clamps to N=20,000 and rebuilds on CPU, and persists the choice; (3) reloading in the *same* browser session (persistence is per-profile, so this specifically needs one continuous session, not separate driver invocations) respects the stored "stay on CPU" preference — no auto-switch, checkbox stays unchecked. Also confirmed the no-GPU-adapter path (plain headless run): stays on CPU at the normal default N=5,000, checkbox visibly grayed out, no console errors.

### As-built deviations (wall behavior + smaller initial cloud, out of milestone sequence)

Requested directly (not part of the M8+ milestone sequence): a much smaller initial cloud relative to the domain, and a live **Wall Behavior** choice (Bounce/Vanish/Wraparound) on both backends. What looked like a two-constant tweak turned into a real cross-cutting change plus a genuine physics-stability finding worth remembering:

- **`INITIAL_CLOUD_RADIUS_FRACTION`** (`Config.ts`) shrunk from an implicit 0.6 to 0.15 — the cloud now starts at 15% of `domainRadius` instead of 60%, giving gravity real room to pull the cloud together before any wall interaction. Both backends' init kept in sync (`fillUniformBall` call in `CpuBackend`, the equivalent Box-Muller ball-sample math in `GpuBackend.initKernel`).
- **`wallBehavior: 'bounce' | 'vanish' | 'wraparound'`** added to `SimParams`, with a Tweakpane dropdown in the Simulation folder. `bounce` is the pre-existing fold-back reflection; `wraparound` reappears on the opposite face with velocity unchanged; `vanish` is the interesting one -- see below.
- **"Vanish" needed a real per-particle mass buffer on the GPU path, not just a rendering trick.** A vanished particle can't simply be teleported off-screen (e.g. beyond the camera's far plane, which is what actually hides it -- see `WALL_VANISH_DISTANCE`'s doc comment) unless it's also excluded from the grid's mass/center-of-mass math first: multiplying a huge coordinate by a *real* mass would corrupt whatever fine/coarse cell it lands in (its clamped grid cell index would still be valid, but the cell's center of mass would get dragged toward that huge coordinate), injecting a large wrong far-field force on any particle whose far-field sum includes that cell -- i.e. most of the cloud. The fix: **mass, not a separate alive flag**. `CpuBackend` already had a `masses` array (always filled with 1, unused for anything but gravity's own math) -- reusing it as a 0/1 alive flag needed zero changes to `gravity.ts`/`UniformGrid.ts`, since both already correctly mass-weight every sum (a mass-0 particle contributes exactly 0, verified, not just assumed). `GpuBackend` had no such buffer at all (M6's docs explicitly noted mass was hardcoded to 1 in every GPU kernel, expected to land with M11) -- this pulled that forward: `GpuUniformGrid`'s `fineAggregateKernel` now sums real per-particle mass instead of assuming every cell member contributes 1, `gpuGravity.ts`'s near-field source term reads real mass instead of a hardcoded constant, and `GpuBackend`'s momentum-correction kernels are now genuinely mass-weighted (dividing by `sum(mass)` instead of raw `count`) so a vanished particle's stale acceleration doesn't skew the correction. Re-verified after this refactor: GPU still matches `computeGridGravity` to float32 precision (`verifyGpuGravity`/`verifyGpuGrid`, ~2e-7 avg relative error, exact grid match) -- the mass-threading didn't regress M6's result.
- **A pre-existing, previously-invisible bug surfaced and got fixed: unbounded step `dt`.** `App.tick()`/`main.ts`'s animation loop passed the raw measured frame time straight to `backend.step()` with no clamp. The very first real animation frame reliably has a large `dt` (~100ms+ measured, vs. the intended ~16.7ms) from page-load/adapter-negotiation overhead before the first callback fires -- harmless with the old, roomier 0.6-fraction cloud, but a single oversized step is enough to destabilize the new denser cloud (semi-implicit Euler with strong, short-range softened gravity isn't unconditionally stable for an arbitrarily large step). Fixed with a `MAX_STEP_DT_MS = 50` clamp in `main.ts`'s render loop, applied only to the physics step -- the *displayed* FPS/CPU/GPU-load monitors still use the real, unclamped frame time. Confirmed directly: without the clamp, a fresh page load's first "real" screenshot a few seconds in showed the cloud already scattered edge-to-edge; with it, the same scenario matches the controlled, manually-stepped reference trace.
- **Softening had to be retuned, and the retuning revealed the dynamics are genuinely stochastic, not a clean function of N.** The smaller cloud packs the same particle count into much less typical spacing, and the old softening (0.08, tuned for the roomier cloud) let close encounters reach escape velocity within the first few frames -- a real, chaotic few-body phenomenon (astrophysically "violent relaxation"), not a bug in the new wall-behavior code. Tuning this by feel was actively misleading: 0.2 looked calm on one random trial, then caused ~93% of particles to escape past r=3 on the next three fresh (differently-seeded) trials -- **a single trial cannot be trusted** for this kind of chaotic system. 0.3 held up consistently at N=5,000 (repeated fresh trials, plus a 900-step/15-simulated-second run staying flat at RMS radius ~0.6-0.7) and became the new default -- but is **not** a universal fix: the same 0.3 exploded at N=15,000, and even 0.5 calmly settled at both N=15,000 and N=100,000 while exploding at N=50,000. No fixed softening value tested eliminates this across the GPU path's full adaptively-benchmarked N range (M7's benchmark regularly picks tens of thousands to 100k+, well above where 0.3 was tuned). Decided (with the user) to accept this tradeoff rather than over-soften the common case: 0.3 is reliable at the CPU path's typical N and is what every session starts from; an occasional violent GPU-path run at a high adaptively-chosen N is a known, accepted possibility worth remembering if it comes up again, not a bug to keep chasing with an ever-bigger constant.
- **Verified on both backends, real GPU hardware included**: `bounce` (existing, re-confirmed), `wraparound` (position wraps, `maxAbs` stays exactly bounded within `domainRadius`, zero NaNs over 180+ steps on both CPU and GPU), and `vanish` (particles correctly stop contributing mass and get parked past the camera's far plane -- confirmed via `pos.x > 4000` counts matching expected vanish behavior, zero NaNs, `coarseMassTotal` still exactly equal to alive-particle count in `verifyGpuGravity`'s grid check).

## Per-stage physics design

- **Stage 1 (gravity)**: softened Newtonian gravity, `F = G·m1·m2·r̂/(|r|²+ε²)^1.5` (Plummer softening), ε exposed as a UI multiplier. *(Done — semi-implicit/symplectic Euler integration, not yet stated explicitly in the original plan but confirmed to conserve energy well over long runs.)*
- **Stage 2/3 (collisions)**: impulse-based sphere-sphere response along the collision normal, generalized with a restitution coefficient `e` (elastic = 1, tunable down for stage 3). **Race-safety detail**: two particles' threads both discover the same pair independently; each thread reads *both* particles' previous-frame state (safe, read-only) but writes only its own new velocity into a separate write buffer, ping-ponging buffers each frame — this is what makes Newton's-third-law-consistent collision response possible without any float atomics. "Merge on contact" avoids GPU stream compaction (a hard parallel algorithm) by using a per-particle `alive` flag instead; dead particles are skipped by physics and scaled to zero size in rendering, with a "reset" button since dead slots aren't reclaimed mid-session.
- **Stage 4 (thermodynamics)**: energy lost per inelastic collision becomes internal energy/temperature on the colliding pair. Radiative cooling uses the **exact closed-form solution** of `dT/dt = −kT⁴`, i.e. `T(t) = T₀ / (1 + 3kT₀³t)^(1/3)`, which is unconditionally stable for any timestep/temperature (unlike explicit-Euler on `T⁴`, which can blow up). Visualized with a small hand-authored 4–5 stop blackbody-ish color ramp (dark red → orange → yellow-white → blue-white).
- **Stage 5 (dark matter)**: one shared particle buffer, partitioned into two contiguous index ranges (baryonic, dark matter) rather than per-particle branching everywhere. Gravity runs over all particles unconditionally (both species have mass). Collision/thermal kernels wrap their body in a species check and skip any dark-matter neighbor, so dark matter is gravity-only with no second physics pipeline. Rendering uses two draw calls (two `Sprite`s) reading the same buffer via an index offset, so each species can be independently colored/toggled.
- **Derived stats (enclosing radius, rotation axis/speed)**: rather than a full GPU parallel-reduction pass (a genuinely harder GPU topic), do a **periodic (~4–6 Hz) CPU-side readback** via `renderer.getArrayBufferAsync()` and compute in plain JS: mass-weighted RMS radius from center of mass; total angular momentum `L = Σm(r−r_com)×v`; inertia tensor `I`; solve `ω = I⁻¹L` (trivial 3×3 inverse). Displayed as a numeric readout plus a wireframe sphere + `ArrowHelper` gizmo in the scene. A full GPU reduction is a documented future optimization if readback cadence ever becomes a bottleneck — not built upfront.

Sequenced so a visible result appears as early as possible, and so each algorithm is learned in plain TypeScript before being ported to TSL/GPU:

5. **M4** ✅ — CPU gravity via the fine+coarse uniform grid at CPU-interactive N. 7. **M6** ✅ — Port grid-gravity to TSL at N=100k–250k on GPU. As built: a from-scratch TSL counting-sort + near/far-field gravity port (no local reference example existed to match against, unlike M5). Verified on real GPU hardware: matches `computeGridGravity`'s CPU output to float32 precision after fixing a real bug in the verification harness itself (a vec3 storage-buffer padding gotcha, not a kernel bug) — see As-built deviations above. Measured ~41ms/step at N=100,000, ~294ms/step at N=250,000.
8. **M7** ✅ — Adaptive-N startup benchmark + runtime GPU⇄CPU toggle with N re-clamping. As built: the benchmark runs once at startup (not on every manual toggle) and picks a default N targeting ~30fps from a real timed measurement; the GPU/CPU choice itself persists across sessions via `localStorage`, defaulting to "use GPU if available" only on the very first run. See As-built deviations above.
9. **M8** — Elastic collisions: CPU first (checked against M4), then GPU (with the ping-pong velocity buffer).
10. **M9** — Inelastic collisions + merge-on-contact, both backends.

## Risks / open questions

- **Binary GPU gate**: no WebGL2 compute fallback exists at all — must detect `navigator.gpu` and force CPU-only gracefully rather than assuming WebGPURenderer alone guarantees a working compute path.
- **Single-thread prefix-sum pass** in the grid build could bottleneck at very large N; a parallel (Hillis-Steele/Blelloch) prefix sum is the documented future optimization, not built upfront.
- **Coarse grid resolution scales with N** (as-built, see deviations above), which means far-field cost isn't perfectly O(N) — grows a bit worse than linear. Carried into the GPU port (M6) unchanged; may want to revisit if N=100k+ far-field cost becomes the bottleneck.
- **Merge-on-contact** doesn't reclaim buffer slots within a session — needs a "reset" affordance (the Restart button covers this today by regenerating the whole cloud, but a lighter per-particle revive isn't built).
- Exact current TSL API names/signatures should be spot-checked against the installed `three` package at the start of each milestone that uses new API surface, since this library moves quickly between releases.
- **TypeScript is pinned to `^6.0.3`** — 7.x's native compiler crashes on this project's TSL usage (see M5 as-built deviations). Don't bump the major version without re-verifying `tsc -b` actually completes on real TSL code, not just a trivial file.
- **The GPU path can now be checked on real hardware from this environment** — headless Chromium fails WebGPU adapter negotiation (M0/M5's original caveat), but a headed browser window on the same machine gets a real adapter (`HEADED=1` in `driver.mjs`, see M6 as-built deviations). M6's gravity kernel is now confirmed to match `computeGridGravity` to float32 precision (avg relative error ~2e-7 at N=500/2,000) via a real cross-backend parity harness (`src/debug/verifyGpuGravity.ts`) — a meaningfully stronger check than the earlier RMS-radius/screenshot pass, which turned out unable to catch a real bug (see M6 as-built deviations' vec3-padding story). `GpuParticlePoints`'s render path specifically hasn't been re-confirmed since M5 beyond "renders a sphere that looks right in a screenshot."
- **Two serial single-thread GPU passes** (`GpuUniformGrid`'s prefix sum, `GpuBackend`'s momentum reduction — both M6) are a confirmed, *measured* performance bottleneck, not just a theoretical one: ~41ms/step at N=100,000 (comfortably interactive) vs. ~294ms/step at N=250,000 (~3.4fps, not interactive), measured via `window.__debug.debugRebuild()` on real hardware. A parallel (Hillis-Steele/Blelloch) version of one or both is the natural next optimization if 250k-scale interactivity matters — not built upfront per this project's stated priorities, but no longer a documented future concern without measurement behind it.
- **Any future debug tooling that reads/writes GPU `vec3` storage buffers as raw `Float32Array`s must account for WGSL's stride-4 padding** (see M6 as-built deviations and `src/debug/vec3Buffer.ts`) — this bit M6's own verification harness hard enough to produce a ~140% average "error" that looked like a real physics bug for a while. Use `unpadVec3()` for reads; for writes, write tightly-packed data and let three's own `updateAttribute()` path re-derive the padded layout (do **not** pre-pad before writing — see `GpuBackend.writeVec3Buffer`'s doc comment for why that double-corrupts).

## Verification

- Each milestone runs via `npm run dev` and is checked visually in the browser for the specific new behavior it adds, **and** numerically via the `window.__debug` hook (exposes `{ app, params, monitors }` for `eval`-based inspection, reach the active backend via `app.getBackend()` — see `.claude/skills/run-particles-simulator/SKILL.md`). M4 specifically showed that visual-only or numeric-only checks each missed a real bug the other caught — use both.
- **Vitest** covers the pure-math, backend-agnostic pieces that don't need a GPU context: `src/sim/cpu/physics/gravity.test.ts` checks grid gravity against the brute-force reference and guards against long-run energy drift. More to add as physics stages land: collision restitution/momentum, the inertia-tensor/angular-velocity solve.
